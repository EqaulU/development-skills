# 開発スキルの「プロジェクト適応」生成プロンプト（汎用）

> 汎用スキル集（development-skills）を **任意のプロジェクト向けにチューニングし、配布可能なスキルパッケージを生成させる** ためのプロンプト。
> 実プロジェクトで行った調整（オーケストレータ skill 化・汎用 skill のプロジェクト適応・不要 skill 除外・移植性確保・既存資産は参照のみ）を、他プロジェクトに再適用できるよう一般化したもの。

---

## 使い方

1. **対象プロジェクトのリポジトリ**で Claude Code を新規セッションで開く（設計資産・コーディング規約・実装コードを Read できる環境）。
2. 末尾の「## 実行用プロンプト」を**まるごとコピー**し、`{{...}}` を実プロジェクトの値に置換（不要行は削除）。
3. Claude Code に貼り付けて実行。Claude が情報収集→確認→生成を行い、`<project>/claude-skills/` 配下に配布パッケージを作る。

> 出力先・命名・配布形態（zip 等）は実行中に確認される。判断が要る所（format、除外 skill、commit 運用）は質問される。

---

## 置換する情報

| プレースホルダ | 内容 | 例 |
|---|---|---|
| `{{プロジェクト名}}` | プロダクト名 | `ShopApp` / `MyProduct` |
| `{{オーケストレータ名}}` | 生成する主 skill 名（kebab-case） | `feature-implement` |
| `{{実装リポジトリと技術スタック}}` | repo 名＋言語/FW（複数可） | `webapp`=Python/Flask, `mobile`=Flutter/Dart |
| `{{設計資産の場所}}` | 仕様・設計書・規約のルート（別repo/SVN可）と、起動時の指定方法 | 別repo `PROJECT_DOCS`（SVN）。プロンプトでパス指定 |
| `{{課題管理}}` | 課題トラッカーと参照方法 | Backlog `PROJECT_KEY`（API/キー） |
| `{{コーディング規約の場所}}` | 規約ドキュメントのパス | `…/コーディング規約/{共通,Web,スマホ}.md` |
| `{{既存のAI/運用資産}}` | 既存のテスト生成プロンプト・レビュー運用・テンプレ等 | `ai-native/docs/prompts/`、セルフレビュー運用 |
| `{{チーム運用}}` | ブランチ/コミット/レビューの慣習 | `feature/KEY`→PR→develop、コミットは自由形式日本語 |
| `{{汎用スキル集の場所}}` | development-skills の場所 | `…/テンプレ/development-skills` |
| `{{スコープ}}`（任意） | 対象機能/リリース範囲 | 次期リリース範囲 |

## 設計原則（プロンプトが守らせること）

1. **オーケストレータ + 汎用 skill の二層**。プロジェクト固有知識は1つの主 skill（`{{オーケストレータ名}}`）に集約し、汎用 skill（development-skills 由来）をフェーズ別に呼ぶ。
2. **format は Skill を既定**。常時読み込みの CLAUDE.md には載せない（関心分離・context 肥大回避）。リポジトリ既存の CLAUDE.md は壊さない。
3. **設計資産は参照、複製しない**。別repo/SVN にある設計書・規約・既存プロンプトはルートパス経由で参照。大きい/バイナリ資産は同梱しない。
4. **移植性**: パッケージに**マシン固有の絶対パスをハードコードしない**。設計資産ルートは「起動プロンプトで指定」を最優先、無ければ上位探索、最後にユーザー確認。
5. **既存運用を尊重**。ブランチ/コミット/レビューの慣習に新ルールを持ち込まない。衝突する汎用 skill（例: commit-msg が Conventional Commits を強制）は採用しない or 書き換える。
6. **不要 skill は同梱しない**。技術スタック上「使用可能性ゼロ」のもの（例: TS無し→type-strengthen、単一言語→i18n-check）は除外。実装ガイドが参照する skill だけに絞る。
7. **必須ワークフロー**: 実装は「計画→（承認/修正）→実装→セルフコードレビュー(MD出力)→修正→再レビュー→単体テスト仕様書」をゲート付きで踏ませる。既存のテスト生成/レビュー運用があればそれに接続。
8. **仕様の正否**: 実装を一律に正としない。設計書/要件を正とし、矛盾は AskUserQuestion で確定。個人情報・権限はログ/URL/エラー露出を点検。

- `<project>/claude-skills/`（配布ステージング）
  - `{{オーケストレータ名}}/SKILL.md`（＋ `references/` に実装ガイド等のプロジェクト固有ドキュメント）

## 実行用プロンプト

> 以下を AI エージェント（Claude Code 推奨）に渡す。`{{...}}` を必ず置換し、不要行は削除すること。

````markdown
あなたは開発支援ツールの構築担当です。汎用スキル集（development-skills）を **このプロジェクト向けにチューニングし、コピーするだけで使える「スキルパッケージ」を生成** してください。

## プロジェクト前提
- プロジェクト名: {{プロジェクト名}}
- 実装リポジトリ/技術スタック: {{実装リポジトリと技術スタック}}
- 設計資産（仕様・設計書・規約）の場所: {{設計資産の場所}}（**このパスは固定せず、本プロンプトで指定された場所を最優先で参照すること**）
- 課題管理: {{課題管理}}
- コーディング規約: {{コーディング規約の場所}}
- 既存のAI/運用資産（あれば活用・複製しない）: {{既存のAI/運用資産}}
- チーム運用（変えない）: {{チーム運用}}
- 汎用スキル集 development-skills の場所: {{汎用スキル集の場所}}
- 対象スコープ（任意）: {{スコープ}}
- 生成する主スキル名: {{オーケストレータ名}}

## 手順

### フェーズ0: 調査（着手前。読むだけ）
1. コーディング規約・既存のAI/運用資産・代表的な実装コードを Read し、**命名/レイヤ/エラー処理/ログ/秘密情報管理/コミット規約/レビュー運用**を把握する。あわせて**成果物・ドキュメント・プロセスの社内呼称**(例: 「方式設計書」「基本設計」「IF仕様書」「DB設計書」「ADR/決定記録」「脅威分析」)を拾う — スキルの `description` 起動キーワード調整に使う。
2. development-skills フォルダを一覧し、**存在する汎用スキル名**を把握する。
3. 設計資産・課題管理の構造（どこに何があるか）を把握する。矛盾/未確定はメモする。
4. 把握結果を要約し、**判断が必要な点はこの後 AskUserQuestion で確認**する（format、除外する skill、commit-msg の扱い、配布形態）。

### フェーズ1: オーケストレータ skill を作る（`{{オーケストレータ名}}/SKILL.md`）
- **土台**: 汎用の `dev-workflow`（要件定義→設計→製造→レビュー→単体/結合/総合テストのゲート付きフロー）を下敷きにし、そこへプロジェクト固有の情報源・規約・課題管理・テスト運用を注入する。ゼロから書き起こさない。
- frontmatter（name / 起動キーワードを含む description）。
- **設計資産ルートの解決順**: ①起動プロンプトでの指定を最優先 → ②実装リポジトリの上位階層を探索 → ③ユーザーに確認。**絶対パスはハードコードしない**。
- **鉄則**: 情報源の優先順位（要件→設計→流れ／一次資料を直読）、実装を一律に正としない、矛盾は AskUserQuestion、コーディング規約遵守、個人情報・権限の露出点検。
- **必須ワークフロー（ゲート付き）**: 事前調査 → 実装計画提示 → （計画OKまで実装しない）→ 実装 → セルフコードレビューをマークダウン出力 → 修正 → 再レビュー（Must Fix が残るうちは完了不可）→ 単体テスト仕様書作成（既存のテスト生成プロンプトがあればそれを使う）→ サマリ報告。
- **フェーズ別に呼ぶ汎用スキル**のマッピング（計画/DB/実装/ログ・セキュリティ/検証/報告）。
- **既存運用への乗せ方**（ブランチ/コミット/レビューは現行どおり。独自ルールを持ち込まない）。
- プロジェクト固有の詳細（スコープ・要件マップ・矛盾一覧など）は `references/` に分離（Progressive Disclosure）。設計資産自体は複製せず参照。

### フェーズ2: 汎用スキルをプロジェクト適応する
- 採用する汎用スキルを `claude-skills/` 直下にコピー。
- **`code-review/references/project-rules.md`**: このプロジェクトの実際のコーディング規約（命名・行数/桁数・秘密情報・通知手段・レイヤ・言語別の具体ルール）で**全面記入**。
- **`pii-check` / `env-check` / `log-design` / `naming-review`** 等の JS/TS 前提スキル: 末尾に「{{プロジェクト名}} 固有」節を追記し、実スタック（言語/FW/シークレット管理/ログ基盤/個人情報定義）に読み替える。
- **除外**: 技術スタック上「使用可能性ゼロ」のスキル（例: TS未使用→type-strengthen、単一言語→i18n-check）は同梱しない。コミット規約と衝突する `commit-msg` は不採用（または現行スタイルに書き換え）。
- **`description` の起動キーワード調整**: 採用する各スキルの frontmatter `description` 末尾に、フェーズ0で把握した**チーム固有の呼称・同義語**を追記して発動精度を上げる（本文は変えず、トリガ語だけ足す）。特に設計系は社内の呼び方に揺れが大きいので必ず確認する:
  - `requirements` … 「要件定義書」「要求仕様」「機能仕様」「受入条件」等
  - `design-doc` … 「方式設計書」「基本設計」「詳細設計」「設計レビュー」等
  - `adr` … 「決定記録」「技術選定メモ」「アーキ決定」等
  - `api-design` … 「IF仕様書」「インターフェース設計」「API仕様」等
  - `data-model` … 「DB設計書」「テーブル設計」「ER設計」等
  - `threat-model` … 「脅威分析」「セキュリティ設計レビュー」「リスクアセス」等

  社内に確立した呼称が無ければ既定の description のままにする（無理に足さない）。追記した語は フェーズ3 のカスタマイズ一覧に記録する。
- 最終的に **オーケストレータが参照するスキルに厳選**する。

### フェーズ3: 移植性のあるパッケージ化
- `claude-skills/README.md` を作る: 構成、インストール（実装リポジトリの `.claude/skills/` へコピー。repo パスは変数指定）、コピー後の対応（設計資産ルートの解決、必要ツール、新セッションで認識）、別PC/zip 配布手順、適用済みカスタマイズ一覧（上流再同期で消えるので再適用できるよう明記）。
- **絶対パス・中間フォルダ名をハードコードしない**。例示は `{{...}}` か `<...>` のプレースホルダにする。
- 既存リポジトリの `.gitignore`/`.claude` 追跡状況を確認し、個人運用かチーム共有かを選べるよう案内。
- 任意で zip を作成（`Compress-Archive`。中身は `claude-skills/` 配下）。

### フェーズ4: 報告
- 生成物一覧（clickable リンク）、採用/除外したスキルと理由、プロジェクト適応の要点、残作業を要約。
- 機微情報（認証・接続先・インスタンスID）は**パッケージに転記せず**所在の参照に留める。

## 厳守事項
- 既存ファイル（リポジトリの規約・テンプレ・設計資産）を実装都合で書き換えない。矛盾は提起する。
- 判断が要る分岐（format、除外スキル、commit 運用、配布形態、設計資産ルート）は**勝手に確定せず確認**する。
- マシン固有の絶対パスをパッケージに残さない。
````

## カスタマイズ判断ポイント（実行前に決める/プロンプトに反映）

- **設計資産が同一リポジトリ内か別repo/SVNか** … 別なら「参照のみ・ルートはプロンプト指定」。同一なら相対参照でよい。
- **既存のテスト生成/レビュー運用の有無** … あれば必須ワークフローの該当ステップを既存資産に接続（無ければ汎用 `test-gen`/`code-review` で代替）。
- **コミット規約** … Conventional Commits 運用なら `commit-msg` 採用可。自由形式なら不採用。
- **配布範囲** … 個人運用（`.claude` 追跡外にコピー）／チーム共有（commit）／zip 配布。
- **ドキュメント/成果物の社内呼称** … 設計書・ADR・API/DB 仕様などの呼び方（方式設計書/基本設計/IF仕様書/DB設計書 等）を、各スキルの `description` に同義語として反映するか。チーム用語が固まっているほど発動精度が上がる。

> このプロンプト自体を development-skills 側に置けば（例: `development-skills/PROJECT_ADAPTATION.md` や `project-adapt` skill）、新規プロジェクトごとに再利用できる。
