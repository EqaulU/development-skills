# 開発フロー プレイブック(詳細)

`dev-workflow` skill 本体から呼び出される、各フェーズのゲートチェックリスト・規模別の軽重・横断観点・戻り方の詳細。

## フェーズ別 ゲートチェックリスト

「次のフェーズに進んでよいか」を判定する完了条件。満たさないうちは進めない。
各フェーズは3ステージ(**①要件定義 / ②設計 / ③実装以降**)に属し、ステージ境界(1→2、2→3、完了前)は**ユーザー承認ゲート**。段の中はゲートを満たせば自走する。

### 1. 要件定義
`requirements` スキルで以下を満たす成果物を作る。
- [ ] 解決したい課題と対象ユーザーが言語化されている
- [ ] **受入基準が検証可能**(「速く」ではなく「p95 200ms 以下」)
- [ ] スコープ(やること)と非目的(やらないこと)が明確
- [ ] 制約(期限・既存システム・規約・コンプライアンス)を把握
- [ ] → 内容をユーザーに提示し **承認を得た**

### 2. 設計
- [ ] 規模に応じた設計スキルを実施(下の right-size 表)
- [ ] 主要な技術判断は `adr` で記録した
- [ ] 代替案を検討し、選定理由が書かれている
- [ ] 横断的関心事(セキュリティ/性能/可観測性/互換性)を検討
- [ ] Open Question が解消、または対応方針が決まっている
- [ ] → 設計をユーザーに提示し **承認を得た**

### 3. 製造
- [ ] 設計どおりに実装されている(逸脱があれば設計へ戻す)
- [ ] 既存のコーディング規約・命名・レイヤに従っている
- [ ] 秘密情報をハードコード/ログ出力していない
- [ ] ビルド・型チェック・Lint が通る

### 4. セルフレビュー①
- [ ] `code-review` を実施し、Must / Should / Nit に分類
- [ ] 必要に応じ深掘り(`security-check` / `perf-review` / `pii-check` 等)
- [ ] 指摘に「なぜ」と修正案が添えられている

### 5. 対応
- [ ] **Must Fix をすべて解消**(Should/Nit は優先度判断)
- [ ] 修正で新たな問題を持ち込んでいない
- [ ] 構造的な問題は `refactor` で整理(挙動は保持)

### 6. セルフレビュー②
- [ ] 再 `code-review` を実施
- [ ] **Must Fix が残っていない**(残るうちは完了不可。4へ戻る)

### 7. 単体テスト
- [ ] `test-gen` で主要ロジックのユニットテストを生成
- [ ] `edge-case-finder` で境界・異常系を補強
- [ ] 正常/境界/異常/例外を網羅、テストが緑

### 8. 結合テスト
- [ ] `integration-test` で継ぎ目(API↔サービス↔実DB、外部APIスタブ)を検証
- [ ] 副作用(DB状態・送信メッセージ)まで確認、テストが緑

### 9. 総合テスト
- [ ] `e2e-scenario` で受入シナリオ(Happy/Alternative/Error)を作成
- [ ] **要件定義の受入基準を満たす**ことを確認、シナリオが緑

### 完了処理(ユーザー承認後のみ)
- [ ] 成果物・残課題をサマリ報告
- [ ] 必要なら `commit-msg` / `pr-description` / `changelog-writer`
- [ ] コミット/プッシュ/PR は **ユーザーの指示と既存規約に従う**(勝手に実行しない)

## 規模に応じた調整(right-size)

| 規模 | 例 | 要件 | 設計 | レビュー | テスト |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 小 | バグ修正、文言変更、軽微な改修 | 一文で意図確認 | 省略可(判断が要れば `adr` 1件) | `code-review` | 回帰の単体テスト |
| 中 | 機能追加、エンドポイント追加 | `requirements` で受入基準を箇条書き | `design-doc` 簡易 + 必要なら `api-design`/`data-model` | `code-review` + 深掘り1-2 | 単体 + 結合 |
| 大 | 新サービス、破壊的変更、認証/決済 | `requirements` でしっかり定義 | `design-doc` + `adr` + `api-design` + `data-model` + `threat-model` | 全深掘り | 単体 + 結合 + 総合 |

判断に迷う中間規模は AskUserQuestion でユーザーに軽重を確認してよい。過剰なセレモニーは速度を奪い、過少は品質を落とす。

## 横断的観点をどこで差し込むか

| 観点 | スキル | 差し込むフェーズ |
| :--- | :--- | :--- |
| 個人情報・機密 | `pii-check` | 設計 / 実装 / レビュー(扱う変更があるとき) |
| ログ設計 | `log-design` | 設計 / 実装 |
| 環境変数・シークレット | `env-check` | 製造 / 完了前 |
| 依存パッケージ | `dependency-audit` | 設計(新規導入時) / 完了前 |
| DBスキーマ変更 | `migration-plan` | 設計 → 製造(反映)→ 結合テストで互換確認 |
| 国際化 | `i18n-check` | 製造 / レビュー(多言語対象のとき) |

## 問題が出たときの戻り方

下流で前提の誤りが判明したら、その場しのぎで直さず**原因のフェーズへ戻す**。

- 結合/総合テストで設計の欠陥が出た → **設計(2)へ戻す**。設計を直してから再実装
- レビューで要件の取り違えが判明 → **要件定義(1)へ戻す**。ユーザーに再確認
- 実装中に設計の不備に気づいた → 設計を更新し、`adr` に判断を追記してから続行

戻る判断と理由をユーザーに伝える。黙って下流で force fix しない。

## ユーザー承認ポイント(明示的に止まる場所)

1. **要件定義の完了後** — 何を作るかの合意
2. **設計の完了後** — どう作るかの合意(ここまで実装に入らない)
3. **完了処理の前** — コミット/PR/リリースの可否

それ以外のフェーズ遷移は、ゲートを満たせば自走してよい(逐一確認で止まりすぎない)。
