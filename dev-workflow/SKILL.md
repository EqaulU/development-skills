# Dev Workflow(開発フロー・オーケストレータ)

要件定義から総合テストまでを、各フェーズで適切なスキルを呼び出しながら、**ゲート(前段の完了条件)を満たすまで次に進まない**形で一気通貫に進める司令塔。

## いつ使うか

- 機能開発を最初から最後まで通して進めたいとき
- 「設計だけ」「テストだけ」でなく、要件→設計→実装→レビュー→テストの全体を回したいとき
- 抜け(設計せず実装/レビューせずマージ/テストせず完了)を仕組みで防ぎたいとき

単一フェーズだけなら個別スキルを直接呼ぶ(例: レビューだけなら `code-review`)。本スキルはそれらを束ねて順に駆動する。

## フロー全体

```
[ステージ1 要件定義] ─承認─▶ [ステージ2 設計] ─承認─▶ [ステージ3 実装以降] ─完了承認─▶ 完了
   requirements          design系を取捨           製造→rev①→対応→rev②→単体→結合→総合
                                                   ▲ 段内は各フェーズにゲート、満たせば自走 ▲
```

3つのステージ境界が**ユーザー承認ポイント**(要件確定後・設計確定後・完了前)。段の中はゲートを満たせば自走し、**段の境界では必ず止まって承認を取る**。問題が出たら原因フェーズへ戻す。

## ステージ構成

### ステージ1: 要件定義 —「何を・なぜ作るか」

| フェーズ | 呼ぶスキル | フェーズのゲート |
| :--- | :--- | :--- |
| 要件定義 | `requirements` | 受入基準が検証可能・非目的が明確 |

→ **段末(承認ゲート)**: 要件をユーザーに提示し、**承認を得てから**ステージ2へ。

### ステージ2: 設計 —「どう作るか」

| フェーズ | 呼ぶスキル | フェーズのゲート |
| :--- | :--- | :--- |
| 設計 | `design-doc` `adr` `api-design` `data-model` `threat-model`(規模に応じ取捨) | 設計レビュー済・未解決点を解消 |

→ **段末(承認ゲート)**: 設計をユーザーに提示し、**承認を得てから**ステージ3へ。**ここまで実装に入らない。**

### ステージ3: 実装以降 —「作って検証する」

段の中は各フェーズのゲートを満たせば自走する(逐一の承認は求めない)。

| # | フェーズ | 呼ぶスキル | フェーズのゲート(次へ進む条件) |
| :--- | :--- | :--- | :--- |
| 3 | 製造 | Claude が実装(既存コーディング規約を遵守) | 設計どおり・ビルド/型/Lint 通過 |
| 4 | セルフレビュー① | `code-review`(深掘り: `security-check` / `perf-review` / `naming-review` / `a11y-review` / `pii-check`) | 結果を Must / Should / Nit で出力 |
| 5 | 対応 | 実装 / 整理が要れば `refactor` | Must Fix をすべて解消 |
| 6 | セルフレビュー② | `code-review` | **Must Fix が残らない**(残るうちは先へ進まない) |
| 7 | 単体テスト | `test-gen`(+ `edge-case-finder`) | 主要ロジックのテストが緑 |
| 8 | 結合テスト | `integration-test` | 継ぎ目(API↔DB / 外部)のテストが緑 |
| 9 | 総合テスト | `e2e-scenario` | 受入基準を満たすシナリオが緑 |

→ **段末(完了承認ゲート)**: 全テスト緑・受入基準充足を報告。**完了処理はユーザー承認後のみ。**

### 完了処理(任意・ユーザー承認後)

報告・コミット/PR に `commit-msg` `pr-description` `changelog-writer`。コミット/プッシュ/PR は**ユーザーの指示と既存規約に従う**(勝手に実行しない)。

各フェーズの詳細なゲートチェックリスト・規模別の軽重・戻り方は [references/playbook.md](references/playbook.md) を参照。

## スキルの呼び出し方

- スキルがあるフェーズは **Skill ツールで該当スキルを起動**する(設計フェーズなら `design-doc` 等)
- スキルが無いフェーズ(**製造・対応**)は本スキルの手順と既存のコーディング規約に従って Claude 自身が実施する
- 横断スキル(`pii-check` / `log-design` / `env-check` / `dependency-audit`)は、該当する変更があるフェーズに差し込む(例: 個人情報を扱うなら設計・実装・レビューで `pii-check`)
- **TodoWrite で各フェーズを ToDo 化**し、進捗とゲート通過を可視化する

## 進め方の鉄則

- **ゲートを越えるまで次へ進まない。** 「設計せず実装」「Must Fix を残してテスト」を許さない。前段未達なら戻る。
- **計画・設計はユーザー承認を取ってから実装する。** 要件定義(1)と設計(2)の完了時に内容を提示し、承認/修正を待つ。勝手に製造に入らない。
- **仕様が正、実装は一律に正としない。** 要件・設計を真とし、実装との矛盾はユーザーに提起して確定する。
- **規模に応じて軽重をつける。** 小さな修正にフルの設計・総合テストは課さない(right-size。playbook 参照)。
- **問題が出たら前フェーズへ戻す。** テストで設計の誤りが判明したら設計に差し戻す。下流での force fix を避ける。
- **コミット/プッシュ/PR は勝手にやらない。** 完了処理はユーザーの指示・既存のブランチ/コミット規約に従う。

## 規模に応じた調整(要点)

| 規模 | 要件 | 設計 | レビュー | テスト |
| :--- | :--- | :--- | :--- | :--- |
| 小(バグ修正・軽微) | 一文で確認 | 省略 or `adr` 1件 | `code-review` | 回帰の単体テスト |
| 中(機能追加) | `requirements` で受入基準を整理 | `design-doc` 簡易 + 必要なら `api-design`/`data-model` | `code-review` + 深掘り1-2 | 単体 + 結合 |
| 大(新サービス・破壊的変更) | `requirements` でしっかり定義 | `design-doc` + `adr` + `api-design` + `data-model` + `threat-model` | 全深掘り | 単体 + 結合 + 総合 |

## 原則

- **オーケストレータは判断と接続に徹する。** 各フェーズの中身は専門スキルに委ね、本体はフェーズ遷移・ゲート判定・承認取得を担う。
- **完了条件を毎フェーズ宣言する。** 「次に進む条件」を明示してから着手し、満たしたことを確認してから進む。
- **既存運用を尊重する。** ブランチ/コミット/レビューの慣習に新ルールを持ち込まない。衝突するスキル(例: `commit-msg` の規約強制)は使わない判断をする。
- **報告は簡潔に。** 各フェーズの結果(成果物・ゲート通過可否・残課題)を短くまとめてから次へ。

## 関連スキル / カスタマイズ

- 各フェーズの個別スキルは上表のとおり。本スキルはそれらの呼び出し順とゲートを定義するだけで、内容は重複させない。
- **プロジェクト固有版**: `PROJECT_ADAPTATION.md` で本スキルを土台に、プロジェクトの設計資産・規約・課題管理・テスト運用を注入した固有オーケストレータを生成できる。汎用の本スキルは中立に保ち、固有知識はそちらへ。
