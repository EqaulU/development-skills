# STRIDE 脅威 × 緩和策カタログ

`threat-model` skill 本体から呼び出されるリファレンス。各 STRIDE カテゴリの典型的な脅威と、設計に落とせる緩和策をまとめる。

## S — なりすまし (Spoofing) / 破られる性質: 認証

| 典型的な脅威 | 緩和策 |
| :--- | :--- |
| 弱い/使い回しパスワード、クレデンシャルスタッフィング | MFA、パスワードポリシー、漏洩パスワード照合、レート制限・ロックアウト |
| セッション/トークンの窃取・固定 | Secure/HttpOnly/SameSite Cookie、短い有効期限、ログイン時の再生成 |
| API キー・サービス間認証の偽装 | mTLS、署名付きリクエスト、鍵ローテーション、最小権限のキー |
| メール/送信元の詐称 | SPF/DKIM/DMARC、送信元検証 |

## T — 改ざん (Tampering) / 破られる性質: 完全性

| 典型的な脅威 | 緩和策 |
| :--- | :--- |
| クライアント送信値(金額・数量・権限フラグ)の改ざん | サーバ側で再計算・再検証。クライアント値を信用しない |
| パラメータ/ヘッダ/Cookie の改ざん | 署名・MAC、サーバ側状態管理、入力検証 |
| 中間者によるデータ書き換え | TLS の強制、証明書検証、HSTS |
| 保存データの不正書き換え | DB 権限分離、変更監査、重要データのハッシュ/署名 |
| 依存パッケージ・成果物の改ざん | 依存の固定(lockfile)、署名検証、SRI、ビルド成果物の検証 |

## R — 否認 (Repudiation) / 破られる性質: 否認防止

| 典型的な脅威 | 緩和策 |
| :--- | :--- |
| 「その操作はしていない」と主張される | 監査ログ(誰が・いつ・何を)、改ざん耐性のあるログ |
| ログの欠落・改ざん | 集中ログ基盤、追記専用、保持期間ポリシー |
| 操作主体が特定できない | 認証済みアクターを全重要操作に紐付け、リクエスト ID 付与 |

## I — 情報漏洩 (Information Disclosure) / 破られる性質: 機密性

| 典型的な脅威 | 緩和策 |
| :--- | :--- |
| IDOR(他人のリソース ID を参照) | リソース所有者・テナントの認可チェックを必須化 |
| エラー/スタックトレースに内部情報 | 本番ではエラー詳細を隠す、汎用メッセージ + サーバ側ログ |
| ログ・URL・分析イベントへの機密情報混入 | マスキング、PII をログに残さない(`pii-check` と連動) |
| 過剰なレスポンス(必要以上のフィールド) | 出力を明示的にホワイトリスト化、シリアライザで制御 |
| 通信の盗聴 | TLS、保存時暗号化、機密は最小限の保持 |
| ディレクトリ/設定/バックアップの露出 | アクセス制御、公開範囲の点検、シークレットを成果物に含めない |

## D — サービス妨害 (Denial of Service) / 破られる性質: 可用性

| 典型的な脅威 | 緩和策 |
| :--- | :--- |
| 大量リクエスト・総当たり | レート制限、スロットリング、WAF、CAPTCHA |
| 重いクエリ・N+1・無制限の結果セット | ページング上限、クエリタイムアウト、コスト見積もり |
| 巨大ペイロード・ZIP 爆弾・正規表現 ReDoS | サイズ上限、解凍上限、安全な正規表現/タイムアウト |
| リソース枯渇(接続・メモリ・ディスク) | コネクションプール上限、バックプレッシャ、クォータ |
| 増幅・依存先の輻輳 | サーキットブレーカ、リトライのジッタ、依存先のタイムアウト |

## E — 権限昇格 (Elevation of Privilege) / 破られる性質: 認可

| 典型的な脅威 | 緩和策 |
| :--- | :--- |
| 一般ユーザーが管理機能を実行 | エンドポイント単位の認可、ロール/権限チェックをデフォルト拒否で |
| 認可チェックの欠落・抜け道 | 集中認可(ミドルウェア/ポリシー)、機能追加時のチェック必須化 |
| パラメータ操作で role/tenant を変更 | サーバ側のセッションから権限を導出、クライアント指定を信用しない |
| 信頼境界をまたぐ内部 API の無防備 | 内部 API も認証・認可、ネットワーク分離だけに頼らない |
| サンドボックス/インジェクションによる実行 | 入力のエスケープ、パラメータ化クエリ、最小権限の実行ユーザー |

## 補助フレーム

- **DREAD**(リスク採点): Damage / Reproducibility / Exploitability / Affected users / Discoverability を各点数化して優先度づけ。簡易には「起こりやすさ × 影響」で十分
- **攻撃面の最小化**: 公開エンドポイント・ポート・権限・依存を減らすほど守る対象が減る
- **多層防御**: 単一の対策に依存しない(例: 認可 + 監査ログ + レート制限)
- **デフォルト安全**: 明示的に許可しない限り拒否。フェイルセーフ(失敗時は安全側に倒す)
