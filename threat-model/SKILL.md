# Threat Model

実装に入る前(または設計レビュー時)に、システムの攻撃面と脅威を洗い出し、緩和策を設計に織り込む。

## いつ使うか

- 新機能・新サービスの設計時(特に認証・決済・個人情報・外部連携を扱う)
- 信頼境界をまたぐデータフローを追加・変更するとき
- 設計レビューでセキュリティ観点を体系的に確認したいとき

`security-check` はコード/diff のレベルで脆弱性を洗う下流の作業。`threat-model` はその上流で、設計段階で攻撃面そのものを評価する。

## 入力情報

開始時に把握する(不明なら質問):

- **対象スコープ** — 分析する機能・サービスの範囲
- **データフロー** — どこから入力が来て、どこを通り、どこに保存されるか
- **信頼境界** — 外部/内部、認証前/後、テナント間など、信頼レベルが変わる線
- **資産** — 守るべきもの(個人情報、認証情報、資金、可用性)
- **アクター** — 正規ユーザー、未認証者、内部者、外部攻撃者

## 進め方

STRIDE 各カテゴリの典型的な脅威と緩和策の対応表は [references/stride.md](references/stride.md) を参照。

1. **対象を図にする** — データフロー図(DFD)を描き、プロセス・データストア・外部要素・データフローを並べる
2. **信頼境界を引く** — 信頼レベルが変わる線を明示(ネットワーク境界、認証境界、テナント境界)
3. **境界をまたぐ点を脅威分析** — 各データフロー・入力点に STRIDE を当てる
4. **脅威を評価** — 起こりやすさ × 影響でリスクを格付け
5. **緩和策を設計に反映** — 対策・受容・移転・回避のいずれかを決め、設計へ織り込む

### STRIDE(脅威カテゴリ)

| 略 | 脅威 | 破られる性質 | 例 |
| :--- | :--- | :--- | :--- |
| S | なりすまし (Spoofing) | 認証 | 他人のトークンで API 呼び出し |
| T | 改ざん (Tampering) | 完全性 | リクエストの金額を書き換え |
| R | 否認 (Repudiation) | 否認防止 | 「操作していない」と主張、ログ無し |
| I | 情報漏洩 (Information Disclosure) | 機密性 | エラーに内部情報、IDOR で他人のデータ |
| D | サービス妨害 (Denial of Service) | 可用性 | 重いクエリ連打、リソース枯渇 |
| E | 権限昇格 (Elevation of Privilege) | 認可 | 一般ユーザーが管理 API 実行 |

## 出力フォーマット

```markdown
## 脅威モデル: <対象機能>

### スコープと前提
- 対象: ...
- 守る資産: 個人情報 / 認証情報 / ...
- 想定アクター: 未認証者 / 認証済み一般 / 内部者

### データフロー図(信頼境界つき)
\`\`\`
[Browser] ──(HTTPS)──┊──→ [API] ──→ [DB]
   未認証              ┊      │
            信頼境界    ┊      └──→ [外部決済API]
\`\`\`
(┊ = 信頼境界)

### 脅威一覧
| # | 箇所 | STRIDE | 脅威 | 起こりやすさ | 影響 | リスク | 緩和策 | 対応 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| 1 | ログインAPI | S | 総当たり/クレデンシャルスタッフィング | 高 | 高 | 高 | レート制限+MFA+ロックアウト | 対策 |
| 2 | 注文取得 | I | IDOR(他人の注文ID参照) | 中 | 高 | 高 | リソース所有者チェック | 対策 |
| 3 | 金額計算 | T | クライアント送信額を信用 | 中 | 高 | 高 | サーバ側で再計算 | 対策 |
| 4 | 検索API | D | 重いクエリの連打 | 中 | 中 | 中 | レート制限+タイムアウト | 対策 |

### 残留リスク
- リスク受容したもの・別途対応に回したものを明記(理由つき)

### 設計への反映
- 上記緩和策のうち設計変更を要するもの(認可チェックの追加箇所、監査ログ設計 等)
```

## 原則

- **信頼境界を起点にする。** 脅威は「信頼レベルが変わる線」をまたぐ点に集中する。まず境界を引く。
- **入力は信用しない。** クライアントから来る値(金額・ID・権限)はすべて改ざん可能と考え、サーバ側で検証・再計算する。
- **STRIDE で網羅的に。** 思いつきでなく、各境界・各入力に 6 カテゴリを機械的に当てて漏れを防ぐ。
- **リスクで優先度をつける。** 起こりやすさ × 影響で格付けし、高リスクから対策する。すべてを同列に扱わない。
- **緩和策は設計に落とす。** 「気をつける」で終わらせず、認可チェック・レート制限・監査ログなど具体的な設計要素にする。
- **受容したリスクも記録する。** 対応しないと決めたものは理由とともに残す。後から「見落とし」と区別できるように。
