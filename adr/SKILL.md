# Architecture Decision Record (ADR)

「なぜこの技術・設計を選んだか」という1つの重要判断を、後から追跡可能な形で記録する。

## ADR とは

アーキテクチャ上の重要な決定を、その文脈・選択肢・結果とともに「1ファイル1決定」で残す軽量ドキュメント。コードからは「何を選んだか」は読めても「なぜ選び、何を捨てたか」は読めない。ADR はそこを埋める。

通常 `docs/adr/NNNN-<タイトル>.md` に連番で保存し、追記のみで運用する(決定が覆ったら新しい ADR を作り、古い方を Superseded にする)。

## いつ書くか

- 後から変えにくい判断(DB、言語/フレームワーク、認証方式、メッセージング基盤)
- 複数の妥当な選択肢があり、トレードオフで選んだとき
- 「なぜ普通と違うやり方なのか」を将来の自分/他人に説明したいとき

設計全体を論じるなら `design-doc`。ADR はその中の個別判断を切り出す単位。

## 入力情報

開始時に把握する(不明なら質問):

- **決めたいこと** — 1つの問い(例:「セッション管理の方式」)
- **制約・前提** — 性能要件、既存構成、チームスキル、期限
- **候補** — 比較する選択肢(最低2つ)

## 出力フォーマット

```markdown
# ADR-0007: セッション管理に JWT ではなくサーバサイドセッションを採用

| 項目 | 内容 |
| :--- | :--- |
| ステータス | Accepted |
| 日付 | YYYY-MM-DD |
| 決定者 | <名前 / チーム> |
| 関連 | ADR-0003, #1234, 設計書リンク |

## 文脈 (Context)
どんな課題・力学(forces)があってこの判断が必要になったか。
制約(性能・セキュリティ・既存構成・期限・チームスキル)を事実ベースで書く。
ここに「決定」は書かない。問題空間だけを書く。

## 決定 (Decision)
何を選んだかを能動態・断定で 1-2 文。
例) ユーザーセッションはサーバサイド(Redis)で管理し、Cookie には署名付きセッションIDのみを格納する。

## 選択肢 (Options Considered)

### 案A: サーバサイドセッション(採用)
- 利点: 即時失効が可能、トークン肥大化なし、漏洩時の影響を限定
- 欠点: ステートフル、Redis の運用コスト

### 案B: JWT(ステートレス)
- 利点: スケールしやすい、外部サービス連携が楽
- 欠点: 即時失効が難しい、ペイロード肥大、鍵運用が複雑
- 不採用理由: 即時失効要件(退会・不正検知時のログアウト)を満たせない

## 結果 (Consequences)
- 良い影響: ログアウト/強制失効が即時。Cookie が軽い。
- 悪い影響・コスト: Redis の可用性が認証の SPOF になる → HA 構成が必要(別途 ADR)。
- 中立: セッションストアの監視項目が増える。

## 備考
- 再検討トリガ: 水平スケールで Redis がボトルネックになった場合
```

## ステータスの遷移

| ステータス | 意味 |
| :--- | :--- |
| Proposed | 提案中。レビュー待ち |
| Accepted | 承認・採用済み |
| Deprecated | もう推奨しないが置き換え先は未定 |
| Superseded by ADR-NNNN | 別の ADR に置き換えられた |

決定を覆すときは既存 ADR を書き換えず、新規 ADR を作って古い方を Superseded にする(判断の履歴を残す)。

## 原則

- **1 ADR = 1 決定。** 複数の判断を詰め込まない。後から個別に参照・更新できる単位に保つ。
- **文脈に決定を混ぜない。** Context は問題空間、Decision は解。混ざると「なぜ」が追えなくなる。
- **却下理由を書く。** 「なぜ採らなかったか」が ADR の核心。将来の蒸し返しを防ぐ。
- **イミュータブルに扱う。** 決定が変わったら上書きせず新 ADR。判断の履歴そのものが資産。
- **短く。** 1ページに収める。長くなるなら設計書(`design-doc`)に分け、ADR からはリンクする。
- **決定時点で書く。** 後追いだと「なぜ」が思い出せない。判断と同時に残す。

## 関連スキル

- `design-doc` — 設計全体を論じる文書。その中の個別判断を ADR に切り出す
- `task-breakdown` — 決定後、実装タスクに分解
- `commit-msg` / `pr-description` — ADR 追加コミット・PR の説明に決定の要約を載せる
