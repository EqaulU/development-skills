# データモデリング パターン集

`data-model` skill 本体から呼び出されるリファレンス。正規形・モデリングパターン・アンチパターン・インデックス判断をまとめる。

## 正規形(RDB)

| 正規形 | 満たす条件 | ひと言 |
| :--- | :--- | :--- |
| 第1正規形 (1NF) | 各列が原子値。繰り返し列・配列を持たない | カンマ区切りで値を詰め込まない |
| 第2正規形 (2NF) | 1NF かつ、非キー列が主キー全体に従属 | 複合キーの一部にしか依存しない列を分離 |
| 第3正規形 (3NF) | 2NF かつ、非キー列が他の非キー列に従属しない | 推移的従属(A→B→C)を排除 |

実務では **3NF を基本**にし、計測された性能要件があるときだけ意図的に非正規化する。

## キー設計

- **サロゲートキー**(自動採番 id): 自然キーが変わりうる/複合になる/外部公開したくないときに有利。結合も軽い
- **自然キー**(業務上の一意値): 意味が安定し、重複防止を DB で強制したいとき。ただし変更に弱い
- 外部公開する ID は連番だと推測・列挙されるため、UUID/ULID や別の公開 ID を検討
- 外部キー(FK)は基本張る。整合性を DB に守らせる。パフォーマンス理由で外す場合は代替の保証を明記

## よくあるモデリングパターン

- **N:M は中間テーブル**: `student` × `course` → `enrollments(student_id, course_id, ...)`。中間テーブルに属性(登録日など)を持てる
- **ポリモーフィック関連**: 「コメントが投稿にも写真にも付く」→ 専用中間テーブル分割か、`commentable_type + commentable_id`。後者は FK が張れない弱点あり
- **状態遷移**: `status` 列 + 遷移履歴テーブル。監査要件があるなら履歴必須
- **ツリー構造**: 隣接リスト(`parent_id`)/ 経路列挙 / 閉包テーブル。読み書きの比率で選ぶ
- **論理削除 vs 履歴テーブル**: `deleted_at` は手軽だが全クエリに条件が必要。監査が重要なら履歴テーブルへ退避
- **金額**: 浮動小数点を使わない。`numeric/decimal` か最小単位の整数(円・セント)で保持
- **列挙**: アプリ enum + DB の CHECK 制約 or 参照テーブル。値の追加頻度で選ぶ

## アンチパターン

- **EAV(Entity-Attribute-Value)**: 何でも `key/value` 行に。柔軟だがクエリ・整合性・性能が破綻しやすい。本当に動的な属性以外は避ける
- **カンマ区切り格納**: `tags = "a,b,c"`。検索も結合もできない。正規化するか配列型/専用テーブルへ
- **巨大な横長テーブル**: 任意項目を 100 列。NULL だらけ。サブタイプに分割を検討
- **意味の二重持ち**: 同じ事実を複数テーブルに重複保持し、片方だけ更新されて不整合
- **過度なフラグ列**: `is_a`, `is_b`, `is_c` が排他なら 1 つの `type` 列に
- **マジックナンバー status**: `status = 3` が何かコードを読まないと分からない。CHECK か参照テーブルで意味づけ

## インデックス判断

引くべきか:
- WHERE / JOIN / ORDER BY / GROUP BY で頻繁に使う列が候補
- カーディナリティが高い(値の種類が多い)列ほど効きやすい
- 真偽値など種類が少ない列の単独インデックスは効きにくい(部分インデックスを検討)

複合インデックスの列順:
- 等価条件(`=`)の列を前、範囲条件(`<`, `>`, `BETWEEN`)の列を後ろ
- `(a, b)` は「a のみ」「a と b」の検索に効くが「b のみ」には効かない(左端プレフィックス則)
- ORDER BY を含めると並べ替えを省ける(`(user_id, created_at DESC)`)

コスト:
- 書き込み(INSERT/UPDATE/DELETE)ごとにインデックスも更新 → 書き込みが重くなる
- 容量を消費する
- **引かれないインデックスは純粋な負債**。アクセスパターンに紐づかないものは作らない

## ドキュメント DB(MongoDB 等)での違い

- 「結合」より「埋め込み(embed)」が基本。一緒に読むものは 1 ドキュメントに
- 1:N で N が無限に増える/単独で巨大になるなら参照(reference)に分ける
- 正規化のメリット(更新の一箇所性)と、埋め込みのメリット(読み取りの局所性)のトレードオフ
- スキーマレスでもアプリ側で「想定スキーマ」を設計・文書化する(無秩序を避ける)
