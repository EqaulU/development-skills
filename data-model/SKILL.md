# Data Model

実装に入る前に、エンティティと関係・キー・インデックスを設計し、データの整合性とアクセス性能の土台を固める。

## いつ使うか

- 新しいテーブル/コレクション群をゼロから設計するとき
- 既存モデルに新しいエンティティ・関係を追加するとき
- アクセスパターンが決まり、それに合うスキーマを選びたいとき

`migration-plan` は既存スキーマの「変更手順」、`sql-optimize` は既存クエリの「チューニング」。`data-model` はその手前で、モデルそのものを設計する。

## 入力情報

開始時に把握する(不明なら質問):

- **ドメイン** — 扱う実体と業務ルール(誰が何を持つ、何回まで、など)
- **アクセスパターン** — 主要な読み書き(何で検索、何で絞り込み、結合の頻度)
- **規模感** — 行数オーダー、増加率、読み書き比率
- **DBMS** — RDB(PostgreSQL/MySQL 等) / ドキュメント / KVS。混在の有無
- **整合性要件** — トランザクション境界、結果整合性の許容範囲

## 設計の観点

正規形・よくあるモデリングパターン・アンチパターン・インデックス判断は [references/patterns.md](references/patterns.md) を参照。

1. **エンティティ抽出** — 名詞を洗い出し、実体と属性に分ける
2. **関係(カーディナリティ)** — 1:1 / 1:N / N:M。N:M は中間テーブル
3. **キー設計** — 主キー(自然キー vs サロゲート)、外部キー、一意制約
4. **正規化** — まず第3正規形を基本に。意図的な非正規化は理由を明記
5. **インデックス戦略** — アクセスパターンから逆算。複合インデックスの列順
6. **データ型と制約** — NOT NULL / CHECK / DEFAULT / 列挙の表現
7. **時系列・履歴** — `created_at`/`updated_at`、論理削除、監査・履歴の持ち方
8. **アクセスパターン適合** — 主要クエリがインデックスで効率的に引けるか

## 出力フォーマット

```markdown
## データモデル: 注文ドメイン

### ER 図
\`\`\`
[users] 1───N [orders] 1───N [order_items] N───1 [products]
                  │
                  └──1:1── [shipments]
\`\`\`

### エンティティ定義

#### orders
| カラム | 型 | 制約 | 説明 |
| :--- | :--- | :--- | :--- |
| id | bigint | PK | サロゲートキー |
| user_id | bigint | FK → users.id, NOT NULL | 注文者 |
| status | text | NOT NULL, CHECK(in enum) | pending/paid/shipped/cancelled |
| total_amount | numeric(12,2) | NOT NULL, CHECK(>= 0) | 合計金額 |
| created_at | timestamptz | NOT NULL, DEFAULT now() | |
| updated_at | timestamptz | NOT NULL | |

#### order_items
(同形式。order_id + product_id に複合一意制約)

### インデックス
| テーブル | インデックス | 目的(対応クエリ) |
| :--- | :--- | :--- |
| orders | (user_id, created_at DESC) | ユーザーの注文一覧を新しい順に |
| orders | (status) where status='pending' | 未処理注文の抽出(部分インデックス) |
| order_items | (order_id) | 注文明細の取得 |

### 設計判断
- サロゲートキー採用(理由: 自然キーが変わりうる/結合が軽い)
- `total_amount` は非正規化(明細から都度集計せず保持。理由: 一覧表示の頻度が高い)
- 履歴は論理削除ではなく `order_status_history` テーブルで監査を担保

### 想定アクセスパターンとの対応
- 「ユーザーの注文履歴(新しい順)」→ orders(user_id, created_at DESC) で充足
- 「未処理注文のバッチ処理」→ 部分インデックスで充足
```

## 原則

- **アクセスパターンから設計する。** 「どう保存するか」より「どう読み書きするか」が先。主要クエリが効率的に引けないモデルは作り直す。
- **まず正規化、非正規化は理由とともに。** 第3正規形を基本にし、性能のための非正規化は「なぜ・どこで整合を取るか」を明記する。
- **キーと制約で整合性を守る。** アプリ層の検証だけに頼らず、NOT NULL・FK・一意制約・CHECK を DB に効かせる。
- **インデックスは万能ではない。** 書き込みコストと容量が増える。引かれないインデックスは負債。アクセスパターンに紐づけて正当化する。
- **将来の変更しやすさを残す。** 列挙のハードコード、過度なフラグ列、意味の二重化(同じ事実を複数箇所)を避ける。
- **DBMS の特性に合わせる。** RDB と ドキュメント DB では正解が違う。結合前提か埋め込み前提かを意識する。
