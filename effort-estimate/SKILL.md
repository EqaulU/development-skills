# Effort Estimate

開発タスクの工数(規模)を、手法を選んで見積もる。単一の数字ではなく、**前提・レンジ・不確実性**をセットで出すのがこのスキルの肝。金額に換算する見積書は `cost-estimate`。

## いつ使うか

- 計画・スケジュール・見積書の前段として工数の桁感を出したいとき
- 「これどのくらいかかる?」を、根拠と幅つきで答えたいとき
- `task-breakdown` で分解した各サブタスクを積み上げて全体規模を出すとき
- `dev-workflow` の計画フェーズ

「何を作るか」が未確定なら先に `requirements`。「どう分けるか」は `task-breakdown`。本スキルはそれらの上に「どのくらいか」を載せる。

## 入力情報

開始時に把握する(不明なら質問):

- **見積もり対象** — タスク/機能の一覧、WBS、要件。粒度はどこまで分解済みか
- **目的と精度** — 概算(桁感)か、コミットメント(確定)か。求められる精度で手法が変わる
- **基準値** — チームのベロシティ、過去の類似実績、1人日の実働時間換算
- **チーム** — 人数、スキル、並列可能性、既存知見の有無
- **不確実性の源** — 未確定要件、未経験技術、外部依存、レビュー/承認プロセス

## 見積もりの進め方

### 1. 手法を選ぶ(詳細は [references/methods.md](references/methods.md))

| 手法 | 向くフェーズ | 特徴 |
| :--- | :--- | :--- |
| 相対見積もり(ストーリーポイント / T-shirt / プランニングポーカー) | 計画〜スプリント | 絶対時間でなく相対比較。アジャイル向き。ベロシティで時間換算 |
| 三点見積もり(PERT) | 個別タスク | 楽観O/最頻M/悲観Pから期待値とばらつきσを算出。不確実性を定量化 |
| 類推(Analogous) | 序盤・粗い | 過去の類似案件から推定。速いが精度は低い |
| パラメトリック | 序盤〜中盤 | 規模ドライバ × 単位生産性(例: 画面数 × 人日/画面) |
| ボトムアップ(WBS積み上げ) | 要件確定後 | `task-breakdown` の各葉を積算。精度高いが時間がかかる |

序盤は類推/パラメトリックで桁感、要件が固まったらボトムアップ、不確実なタスクは三点見積もりで幅を持たせる、という**段階的な精緻化**が基本。

### 2. 不確実性をレンジで表す

- **点見積もり1個では出さない。** レンジ(最小〜最大)か信頼度(例: 70%で5人日以内)を付ける
- **不確実性コーン** — 序盤は実態の ±数倍ぶれる。要件・設計が固まるほど収束する。序盤の数字を確定値として扱わない
- **三点見積もりの合算**: 期待値 E=(O+4M+P)/6、σ=(P−O)/6。全体の期待値は ΣE、全体のσは √(Σσ²)(個別σの単純合計ではない)。計算手順は references/methods.md 参照

### 3. 実装以外の工数を必ず含める

「実装だけ」の見積もりは確実に過小になる。以下を内訳に明示するか、係数で上乗せする:

- テスト(ユニット/結合/E2E)、コードレビュー、指摘対応・手戻り
- 設計・ドキュメント、環境構築、デプロイ、動作確認
- 会議・調整・仕様確認、不具合修正
- 目安: 純粋な実装工数の 1.5〜2 倍に膨らむことが多い(チーム実績で補正)

### 4. バッファは分けて、まとめて持つ

- **コンティンジェンシー予備** — 既知のリスク(未経験技術・外部依存など)に対する備え。リスク単位で積む
- **マネジメント予備** — 未知の未知への備え。全体に対する率で確保
- 各タスクに薄く隠さず、まとめて1つで持つ(学生症候群/パーキンソンの法則で薄いバッファは消える)

## 出力フォーマット

```markdown
## 工数見積もり: ユーザー登録機能

### 見積もり方式
ボトムアップ(task-breakdown のWBSを積み上げ)。不確実なタスクは三点見積もり(PERT)で幅を算出。

### 前提(Assumptions)
- メール送信は既存 SES を流用(新規構築なし)
- 開発者2名、1人日=実働6h換算、レビュー含む
- 既存の認証基盤を再利用できる

### 内訳(三点見積もり)
| ID | タスク | O | M | P | 期待値E | σ |
| :--- | :--- | --: | --: | --: | --: | --: |
| #101 | DBスキーマ追加 | 0.3 | 0.5 | 1.0 | 0.55 | 0.12 |
| #102 | 登録API | 0.8 | 1.0 | 2.0 | 1.13 | 0.20 |
| #103 | 確認API | 0.5 | 0.8 | 1.5 | 0.87 | 0.17 |
| #104 | 登録UI | 0.5 | 1.0 | 1.5 | 1.00 | 0.17 |
| #106 | E2E | 0.3 | 0.5 | 1.0 | 0.55 | 0.12 |
| | **実装系 合計** | | | | **4.10** | **0.36** |

合計σ = √(0.12²+0.20²+0.17²+0.17²+0.12²) ≈ 0.36

### 工程別の上乗せ
| 工程 | 比率/算出 | 人日 |
| :--- | :--- | --: |
| 実装系(上記) | — | 4.1 |
| 設計・ドキュメント | 実装の20% | 0.8 |
| 管理・調整・会議 | 実装の15% | 0.6 |
| 小計 | | 5.5 |

### バッファ
- コンティンジェンシー(メールプロバイダ仕様の確認リスク): +0.5人日
- **最終レンジ: 5.5〜7.0 人日(期待値6.0、信頼度70%で6.4人日以内)**

### スケジュール換算
2名並列・依存関係考慮で、暦日 約4〜5営業日(#101→#102/#103→UI→E2E のクリティカルパス)。

### リスク・確認事項
- メール送信プロバイダの選定が未確定 → 確定するまで上振れ要因
- パスワード強度ポリシー未定 → UI/API の検証実装に影響
- 見積もりは要件確定時点のもの。スコープ追加時は再見積もり
```

## 原則

- **点ではなくレンジで出す。** 単一の数字はコミットメントと誤解される。最小〜最大か信頼度を必ず添える。
- **前提を明示する。** 何を所与としたかが変われば見積もりは変わる。前提が崩れたら再見積もり、と書いておく。
- **実装以外を忘れない。** テスト・レビュー・手戻り・調整・ドキュメントを内訳か係数で必ず含める。
- **段階的に精緻化する。** 序盤の粗い数字を確定値として扱わない。情報が増えたら見積もり直す。
- **過去実績を基準にする。** 勘でなく、チームのベロシティ・類似案件の実績にアンカリングする。
- **バッファは分けて見える化する。** 各タスクに隠さず、リスク予備と管理予備を別建てで持つ。
- **見積もり不能なら調査タスクを切る。** 不確実性が高すぎるものは、いきなり実装見積もりにせずスパイク(調査)を先に置く。

## 関連スキル

- `task-breakdown` — 見積もりの単位となるサブタスクへ分解(本スキルの入力)
- `requirements` — 何を作るかが未確定なら先に確定(前提の精度が見積もりの精度)
- `cost-estimate` — 工数(人日/人月)を金額の見積書に換算(下流)
- `dev-workflow` — 計画フェーズで本スキルを呼ぶ
