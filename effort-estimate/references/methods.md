# 見積もり手法カタログ & 計算チートシート

`effort-estimate` skill 本体から呼び出されるリファレンス。各手法の使い方、PERT/標準偏差の計算、合意形成の進め方、アンチパターンをまとめる。

## 手法ごとの使い方

### 相対見積もり(ストーリーポイント)

絶対時間ではなく「この作業はあの作業の何倍か」で相対的に大きさを表す。人によって作業速度が違っても相対比は安定しやすい。

- **スケール** — フィボナッチ風(1, 2, 3, 5, 8, 13, 20, 40, 100)。大きいほど刻みを粗く=不確実性が大きいことを表現
- **基準ストーリー** — 過去の「3ポイントだったタスク」を基準に据えて相対比較する
- **時間換算** — ベロシティ(1スプリントで消化できるポイント数)で割る。例: 直近3スプリント平均 20pt/スプリント、残り 60pt → 約3スプリント
- **20以上は分割**を検討(大きすぎて見積もり精度が落ちる)

### T-shirt サイジング

XS/S/M/L/XL の粗い分類。ポートフォリオ・ロードマップ段階で、多数の項目をざっくり相対比較するのに使う。後で各サイズに代表ポイント/人日をマッピングして概算化する。

### プランニングポーカー / ワイドバンドデルファイ

複数人で見積もりを収束させる合意形成法。

1. 各自が同時にカード(ポイント)を出す(他人に引きずられないよう同時公開)
2. 最大と最小を出した人が**根拠を説明**する(ここで前提のズレ・見落とした作業が表面化する)
3. 議論後に再投票。数ラウンドで収束させる
4. 収束しなければ「分割が必要」「調査が必要」のサイン

価値は数字そのものより、**認識のズレを炙り出す対話**にある。

### 三点見積もり(PERT)

不確実性のある個別タスクを、3つのシナリオから定量化する。

- **O(Optimistic)** — 全部うまくいった場合の楽観値
- **M(Most Likely)** — 最もありそうな値
- **P(Pessimistic)** — 問題が重なった場合の悲観値

```
期待値 E = (O + 4M + P) / 6
標準偏差 σ = (P − O) / 6
分散 = σ²
```

複数タスクの合算(タスクが独立と仮定):

```
全体の期待値 = ΣE(各タスクの期待値の単純合計)
全体の分散  = Σσ²(各タスクの分散の合計)
全体のσ    = √(Σσ²)   ← σの単純合計ではない点に注意
```

信頼区間の目安(正規近似):

| 範囲 | 含む確率 |
| :--- | :--- |
| E ± 1σ | 約68% |
| E ± 2σ | 約95% |
| E + 1.3σ 以内 | 約90%(片側) |

例: E=4.1人日, σ=0.36 → 「90%の確率で 4.6人日以内」と言える。コミットメント値はEそのものでなく E+kσ で安全側に取る。

### 類推見積もり(Analogous)

過去の類似案件・機能の実績を基準に、規模差で補正する。トップダウンで速い反面、類似性の判断に依存するので序盤の桁感向き。「前回のログイン機能が5人日、今回はSSO対応で1.5倍だから約7.5人日」のように使う。

### パラメトリック見積もり

規模を表すドライバ(数量)と単位あたりの生産性を掛ける。

| ドライバ | 単位生産性の例 |
| :--- | :--- |
| 画面数 | 1画面あたり ○人日 |
| API エンドポイント数 | 1本あたり ○人日 |
| テーブル数 / 移行件数 | 1テーブルあたり ○人日 |
| 帳票数 | 1帳票あたり ○人日 |

単位生産性は**自チームの実績**から出す。書籍やベンダ提供の係数は出発点にしかならない。

### ボトムアップ(WBS積み上げ)

`task-breakdown` で分解した末端タスクを個別に見積もり、合算する。最も精度が高いが、分解が済んでいないと使えず、時間もかかる。末端が大きすぎる(>2〜3人日)なら更に分解してから見積もる。

## 何を見積もりに含めるか(漏れチェック)

実装以外で漏れやすい工数:

- [ ] テスト作成・実行(ユニット/結合/E2E)
- [ ] コードレビューと指摘対応の手戻り
- [ ] 設計・ドキュメント・コメント
- [ ] 環境構築・CI/CD 設定・デプロイ作業
- [ ] データ移行・マイグレーション
- [ ] 仕様確認・調整・会議・承認待ち
- [ ] 不具合修正・リグレッション対応
- [ ] 学習コスト(未経験の技術・ドメイン)
- [ ] リリース作業・リリース後の見守り

## 精度を上げる原則

- **過去実績にアンカリング** — 勘ではなくベロシティ・類似実績を基準点にする
- **小さく割って積む** — 大きい単位の直感より、小さく分けて足すほうが系統誤差が相殺されやすい
- **複数人・複数手法でクロスチェック** — ポーカーで認識を揃え、ボトムアップとパラメトリックの数字を突き合わせる
- **実績との差分を振り返る** — 見積もりと実績の乖離を記録し、次の補正係数にする(`work-log` が原資料に使える)

## アンチパターン

| やりがち | なぜ危険 | 対処 |
| :--- | :--- | :--- |
| 点見積もり1個を提示 | 確定コミットと誤解される | レンジ/信頼度で出す |
| 楽観値だけを採用 | ほぼ必ず超過する | M または E、安全側は E+kσ |
| 各タスクに薄くバッファを隠す | 学生症候群で消える | バッファはまとめて別建て |
| 実装時間だけ数える | テスト・手戻り・調整が抜けて過小 | 工程別係数か内訳で明示 |
| 序盤の概算を確定値扱い | 不確実性コーンを無視 | 段階的に再見積もり |
| 締切から逆算して工数を決める | 願望であって見積もりではない | 見積もりと締切は分け、ギャップはスコープ調整で埋める |
| 見積もり不能なものを実装見積もり | 大ハズレの温床 | スパイク(調査タスク)を先に切る |
