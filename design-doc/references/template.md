## 背景 / Context
なぜ今これに取り組むのか。読者がこの文書を理解するための前提知識。
- 現状の課題: 例) 注文処理が単一 DB に集中し、ピーク時に p95 が 2s を超える。
- きっかけ: 例) 来季のキャンペーンでトラフィック 3 倍を見込む。
- 前提・既知の事実: 既存構成、関連する過去の決定。
ここには解決策を書かない。問題空間だけを書く。

## 目的 / Goals
達成したいことを、できる限り計測可能に書く。
- [ ] 注文処理の p95 レイテンシを 200ms 以下にする
- [ ] ピーク時 3,000 req/s を可用性 99.9% で捌く
- [ ] 既存クライアントの API 互換を維持する

## 非目的 / Non-Goals
今回やらないことを明示する(スコープ膨張の防止)。
- 在庫管理サービスのリファクタは対象外(別 Epic)
- 管理画面のUI改善は含まない
- マルチリージョン対応は将来検討(本書では単一リージョン前提)

## 提案 / Proposed Design
本命の設計。まず全体像、次に詳細。

### 全体像
構成要素とデータの流れを示す。

\`\`\`
[Client] ──→ [API Gateway] ──→ [Order Service] ──→ [Order DB (primary)]
                                      │                     │
                                      └──→ [Cache/Redis]    └──→ [Read Replica]
\`\`\`

### 詳細
- **データモデル**: 主要エンティティ、テーブル/コレクション、主キー・インデックス
- **API / インターフェース**: エンドポイント or 関数シグネチャ、リクエスト/レスポンス、エラーモデル
- **主要なロジック / フロー**: 正常系の処理順序、状態遷移
- **外部依存**: 連携する外部サービス、SLA、障害時の振る舞い(フォールバック)

### 案B: <別案>

### 案C: 何もしない / 現状維持

## 横断的関心事 / Cross-Cutting Concerns
- **セキュリティ**: 認証認可、入力検証、機密データの扱い、攻撃面の変化
- **プライバシー**: 扱う個人情報、保持期間、最小化
- **パフォーマンス**: 想定負荷、ボトルネック、キャッシュ戦略、計算量
- **可観測性**: メトリクス、ログ、トレース、アラート閾値
- **互換性**: 後方互換、API バージョニング、既存データとの整合

## 段階移行 / Rollout
- **リリース手順**: フェーズ分け(例: 内部 → 5% → 100%)
- **フィーチャーフラグ**: 切替え単位、デフォルト値
- **データ移行**: 必要なマイグレーション、Expand-Migrate-Contract の適用有無
- **ロールバック**: 戻し方、戻せない地点(point of no return)はどこか
- **監視**: リリース後に見る指標と異常時の判断基準

## リスクと未解決の問い / Risks & Open Questions
- リスク: 例) Read Replica のレプリケーション遅延で読み取り不整合 → 緩和策: ...
- Open Question: 例) Cache の TTL は何秒が妥当か → レビューで決定 / 計測で決定
- 依存する外部要因: 例) インフラチームの Redis クラスタ提供時期

## テスト方針 / Test Plan
- ユニット: 重点的に検証するロジック
- 結合 / E2E: 検証する主要シナリオ
- 負荷試験: 目標スループット・レイテンシの検証方法
- 移行検証: 本番相当データでのドライラン
