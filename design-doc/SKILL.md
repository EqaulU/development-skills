# Design Doc

実装に入る前に「何を・なぜ作るか」と設計判断をチームで合意するための設計書(RFC)を作る。

## いつ使うか

- 複数人・複数コンポーネントに影響する変更
- 後戻りコストの高い技術選定(DB、フレームワーク、外部サービス、公開 API)
- 「なぜこの設計にしたか」を後から参照したくなる変更

軽微な変更にフルの設計書は不要。1コンポーネントの単一判断なら `adr` のほうが軽い。

## 入力情報

開始時に把握する(不明なら質問):

- **解決したい課題** — 誰の、どんな困りごとか。きっかけ(Issue、要望、障害)
- **制約** — 期限、既存システム、チームスキル、予算、コンプライアンス
- **成功条件** — 何が達成されたら成功か(できれば計測可能な指標)
- **スコープ境界** — 今回やること / やらないこと
- **既存資産** — 流用できる仕組み、過去の関連 ADR / 設計書

## 構成(概要)

以下のセクションを含む Markdown 文書を生成する。フルテンプレートは [references/template.md](references/template.md) を参照。

1. **メタ情報** — タイトル、著者、レビュアー、ステータス、最終更新日
2. **背景 / Context** — なぜ今これに取り組むか。前提知識
3. **目的 / Goals** — 達成したいこと(計測可能に)
4. **非目的 / Non-Goals** — 今回やらないこと(スコープ膨張の防止)
5. **提案 / Proposed Design** — 本命の設計。全体像 → 詳細
6. **検討した代替案 / Alternatives** — 他案と、それを採らなかった理由
7. **横断的関心事** — セキュリティ / プライバシー / パフォーマンス / 可観測性 / 互換性
8. **段階移行 / Rollout** — リリース手順、フラグ、ロールバック、データ移行
9. **リスクと未解決の問い / Risks & Open Questions**
10. **テスト方針 / Test Plan**

## 設計の進め方

- **問題から書く。** いきなり解決策を書かない。「何が困っているか」が曖昧な設計書は議論が発散する。
- **非目的を先に固める。** スコープ外を明示すると、提案も代替案も評価しやすくなる。
- **代替案を必ず2つ以上。** 「これしかない」設計書はレビューにならない。却下した案こそ思考の証拠。
- **可逆性で深さを変える。** 後から変えにくい判断(DB / 公開 API)は厚く、変えやすい判断は薄く。

## 原則

- **「なぜ」を最優先。** どんな設計かは図とコードで分かる。なぜその設計を選び、他を捨てたかを書く。
- **計測可能なゴールを置く。** 「速くする」ではなく「p95 を 800ms → 200ms」。成否を後で判定できるように。
- **非目的を明示する。** やらないことを書かないと、レビューでスコープが膨張する。
- **可逆性を意識する。** 戻せない判断は時間をかけて検討、戻せる判断は早く決めて前に進む。
- **未解決を隠さない。** 決まっていない点は「Open Question」として明示し、レビューで埋める。
- **既存テンプレを尊重。** チームに設計書テンプレ(社内 Wiki 等)があればそのフォーマットに合わせる。

## 関連スキル

- `requirements` — 「何を作るか」を確定する上流。本スキルはそれを「どう作るか」へ展開する
- `adr` — 設計書内の個別の重要判断は ADR として切り出すと、後から決定理由を追いやすい
- `task-breakdown` — 合意した設計を実装サブタスクに分解
- `migration-plan` — DB スキーマ変更を伴う場合の段階移行の詳細
- `api-doc` — 設計した API を実装後に OpenAPI として文書化
- `security-check` — 横断的関心事のうちセキュリティの深掘り
