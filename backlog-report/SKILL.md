# Backlog Report (作業ログ → Backlog 登録)

`work-log` で残した **当日分の作業ログを要約**し、Backlog(Nulab のプロジェクト管理 SaaS)の**ドキュメント / Wiki に API で登録**する。日々の生ログを、チーム/上長が読む日報の形にして共有するためのスキル。

> ここでの「Backlog」は Nulab の SaaS を指す。製品の優先タスク一覧(プロダクトバックログ)の分解は `task-breakdown`。

## いつ使うか

- 1日の終わりに、その日の作業ログを日報として Backlog に上げたいとき
- 複数の `work-log` エントリを1本の要約にまとめて共有したいとき
- 作業サマリを定期的に Backlog ドキュメント/Wiki に蓄積したいとき

要約元が無ければ先に `work-log` でエントリを残す。Backlog 登録をしない純粋な要約だけなら、このスキルの「要約フォーマット」だけ使えばよい。

## 全体の流れ

1. **入力を集める** — 当日の作業ログ `docs/worklog/YYYY-MM-DD.md` を読む。無ければ会話履歴・`git log --since=midnight`・`git diff` から拾う。
2. **要約を作る** — 下記フォーマットで Markdown を生成し、`docs/worklog/summary-YYYY-MM-DD.md` 等に書き出す。
3. **内容を確認** — 生成した要約をユーザーに提示する。
4. **登録する** — 確認が取れてから Backlog API に POST する。投稿は外部公開操作なので**承認を得てから実行**(下記「登録(投稿)」)。

## 入力情報

開始時に把握する(不明なら質問):

- **対象日** — 既定は当日(`YYYY-MM-DD`)
- **作業ログの場所** — 既定 `docs/worklog/YYYY-MM-DD.md`(`work-log` の出力)
- **登録先** — Backlog の**ドキュメント**(既定)か **Wiki** か
- **接続情報** — スペース / API キー / プロジェクト ID(環境変数で渡す。下記「設定」)

## 要約フォーマット

```markdown
# 作業ログサマリ YYYY-MM-DD <担当者>

## TL;DR
- 今日の成果と状態を 1〜3 行で

## 今日やったこと
- <タスク名> — 成果(完了 / 進行中) [#1234]
- ...

## 進捗ステータス
| タスク | 状態 | メモ |
| :--- | :--- | :--- |
| ... | 完了 / 進行中 / 中断 | ... |

## 詰まった点・課題・リスク
- 問題 → 対応(解決 / 未解決)。エスカレが要るものは **要対応** と明記

## 明日の予定 / 次のアクション
- [ ] ...

## 関連
- Issue: #... / PR: ... / 主要コミット: `abc1234`
```

要約の原則:

- **生ログをそのまま貼らない。** 読み手(チーム/上長)が3分で把握できる粒度に圧縮する。
- **状態を必ず示す。** 完了・進行中・ブロック中を取り違えない。ブロックは目立たせる。
- **事実と推測を分ける。** 未確認は「未検証」と書く。
- **機密・PII を載せない。** トークン・パスワード・個人情報・社外秘を要約に含めない(`pii-check` の観点)。Backlog は社外メンバーが入っている場合もある。

## 設定(接続情報)

API キーは**シークレット**。コードや作業ログにベタ書きせずに環境変数で渡す(`env-check` の観点)。置き場所は次のいずれか:

- **OS のユーザー環境変数**(推奨。リポジトリツリーに痕跡が残らない)
- **`.claude/settings.local.json` の `env`**(git 除外される前提。手軽)

> `.claude/settings.json`(`.local` なし)は **git にコミットされる**ため、ここに鍵を書かない。`settings.local.json` が無視対象かは `git check-ignore .claude/settings.local.json` で確認する。`.env` を使う場合は `.gitignore` に `.env` があるか先に確認する。

| 変数 | 例 | 用途 |
| :--- | :--- | :--- |
| `BACKLOG_SPACE` | `yourspace.backlog.jp` | スペースのホスト名(`.backlog.com` / `.backlog.jp` / `.backlogtool.com`) |
| `BACKLOG_API_KEY` | `xxxxxxxx` | 個人設定 > API で発行 |
| `BACKLOG_PROJECT_ID` | `12345` | 登録先プロジェクトの数値 ID |
| `BACKLOG_DOC_PARENT_ID` | (任意) | 親ドキュメント ID(配下にぶら下げる場合) |

## 登録(投稿)

**外部公開操作。要約をユーザーに提示し、承認を得てから POST する。** 初回は dry-run(投稿コマンドを表示するだけ)を推奨。

- 既定は **Document API**(`POST /api/v2/documents`)。当日分を1ドキュメントとして新規作成する。
- 追記・更新したい運用(1ページを育てる)なら **Wiki API**(`PATCH` で更新可)を選ぶ。
- 正確なエンドポイント・パラメータ・curl/PowerShell スクリプト・エラー対処・ドメイン差異は [references/backlog-api.md](references/backlog-api.md) を参照。

投稿後は、返ってきた**ドキュメント/Wiki の URL をユーザーに報告**する。

## 原則

- **承認なしに投稿しない。** Backlog への登録は取り消し・編集が手間。内容確認 → 承認 → 投稿の順を守る。
- **キーを残さない。** API キーは環境変数のみ。ログ・要約・コミットに出さない。
- **冪等性に注意。** Document API は作成/削除中心(更新は当日分を新規作成で運用)。二重投稿を避けるため、再実行前に既存ドキュメントの有無を確認する。Wiki は同名ページを `PATCH` で更新できる。
- **要約は要約に徹する。** 生ログ全文の転記ではなく、判断・成果・次アクションを残す。

## 関連スキル

- `work-log` — このスキルの入力。当日分の作業ログエントリを生成する(対の関係)
- `pii-check` — 要約に PII・社外秘が混ざっていないか確認
- `env-check` — API キー等のシークレットが漏れていないか確認
- `incident-postmortem` — 障害対応日の作業サマリは、そのままポストモーテムのタイムライン素材になる
- `changelog-writer` — リリース時は作業サマリから利用者向け変更を抽出
- `task-breakdown` — 製品バックログ(優先タスク)の分解はこちら。本スキルとは別
