# Requirements

口頭ベースの曖昧な要望を、検証可能な要件・ユーザーストーリー・受入基準に整理する。「何を・なぜ作るか」を確定する上流工程。

## いつ使うか

- 実装・設計に入る前に「何を作るか」を固めたいとき
- 要望が曖昧/口頭ベースで、検証可能な形に落としたいとき
- `dev-workflow` の要件定義フェーズ

設計(どう作るか)は `design-doc`。本スキルはその前段「何を・なぜ作るか」を担い、解決策(how)には踏み込まない。

## 入力情報

開始時に把握する(不明なら質問):

- **背景・課題** — 誰の、どんな困りごとか。なぜ今やるのか
- **関係者** — 利用するユーザー/ロール、ステークホルダー
- **ビジネス制約** — 期限、予算、コンプライアンス、既存システム
- **成功の定義** — 何が達成されたら成功か(できれば指標)

## 要望から要件へ

- **要望(〜したい)を鵜呑みにせず、背後の課題(なぜ)を掘る。** 提示された解決策の奥にある真のニーズを要件にする
- **曖昧語を潰す。** 「速く」「使いやすく」「大量」を数値・具体に(質問バンクは [references/checklist.md](references/checklist.md) 参照)
- **未確定は明示する。** 確定が要るものは質問、仮に進めるものは Assumption として記録

## 要件の種類

- **機能要件** — システムが何をするか(ユーザーストーリーで表現)
- **非機能要件(NFR)** — 性能・可用性・セキュリティ・運用性など。カテゴリと観点は references/checklist.md を参照

## ユーザーストーリーと受入基準

- ストーリー形式: **As a [役割], I want [目的], so that [価値]**
- 粒度は INVEST を意識(独立・交渉可能・価値・見積可能・小さい・テスト可能)。大きすぎれば分割は `task-breakdown` へ
- 受入基準は **Given-When-Then** 形式かチェックリストで、**検証可能**に書く(そのまま `e2e-scenario` の元ネタになる)

## 優先度

MoSCoW で分類する: **Must**(必須)/ **Should**(重要だが代替可)/ **Could**(あれば良い)/ **Won't**(今回やらない=非目的)。

## 出力フォーマット

```markdown
## 要件: カートまとめ買い

### 概要
複数商品を1回の決済で購入できるようにし、都度決済の手間をなくす。

### スコープ
- 含む: カート内商品の一括購入、在庫チェック、確認メール
- 含まない(非目的): 分割配送、定期購入、ゲスト購入(別途検討)

### ユーザーストーリー

**US-1**(優先度: Must)
As a 登録ユーザー, I want カート内の商品をまとめて購入したい, so that 何度も決済せずに済む
受入基準:
- [ ] Given カートに在庫のある商品がある / When 購入を確定する / Then 注文が作成され、在庫が減り、確認メールが届く
- [ ] Given カート内に在庫不足の商品がある / When 購入を確定する / Then エラーが表示され、注文は作成されない

**US-2**(優先度: Should)
As a 登録ユーザー, I want 購入前に合計金額を確認したい, so that 想定外の課金を避けられる
受入基準:
- [ ] Given カートに商品がある / When 確認画面を開く / Then 小計・送料・合計が表示される

### 非機能要件
| 種別 | 要件 |
| :--- | :--- |
| 性能 | 注文確定の p95 < 500ms |
| 可用性 | 営業時間中 99.9% |
| セキュリティ | カード情報を自社で保持しない(PCI スコープ外) |

### 優先度(MoSCoW)
- Must: US-1
- Should: US-2
- Could: クーポン適用
- Won't(今回やらない): ゲスト購入、分割配送

### 前提と未確定(Assumptions / Open Questions)
- 前提: 決済は既存の Stripe 連携を流用する
- 未確定: 在庫はリアルタイム引当か、確定時引当か → 要確認
```

## 原則

- **「なぜ」を掘る。** 要望の裏の課題を要件にする。提示された解決策をそのまま要件化しない。
- **検証可能に書く。** 「速い」「安定」ではなく、後でテストで判定できる数値・条件にする。
- **解決策を混ぜない。** 要件は what と why。how(技術選定・実装方式)は `design-doc` の領域。
- **非目的を明示する。** Won't / スコープ外を書かないと、後から際限なく要望が膨張する。
- **未確定を隠さない。** Open Question と Assumption を分けて残し、確定をユーザーに委ねる。
- **ステークホルダーの言葉で書く。** 非開発者も読む。実装用語(クラス名・テーブル名)を持ち込まない。

## 関連スキル

- `design-doc` — 確定した要件を「どう作るか」へ落とす(下流)
- `task-breakdown` — ユーザーストーリーを実装サブタスクに分解
- `e2e-scenario` — 受入基準(Given-When-Then)を E2E シナリオへ展開
- `dev-workflow` — 要件定義フェーズの担い手。本スキルを最初に呼ぶ
