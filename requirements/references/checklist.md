# 要件定義 チェックリスト & 質問バンク

`requirements` skill 本体から呼び出されるリファレンス。非機能要件のカテゴリ、曖昧さを潰す質問、受入基準の書き方をまとめる。

## 非機能要件(NFR)カテゴリ

機能要件に隠れて漏れやすい。該当するものを確認する。

| カテゴリ | 確認する観点 | 問いの例 |
| :--- | :--- | :--- |
| 性能 | レイテンシ、スループット | 目標応答時間(p95)は? ピーク時の req/s は? |
| 可用性・信頼性 | 稼働率、復旧目標 | 許容ダウンタイムは? RTO/RPO は? |
| スケーラビリティ | 想定負荷、増加率 | 想定データ量・ユーザー数と、その増加見込みは? |
| セキュリティ | 認証認可、暗号化、監査 | 誰が何にアクセスできる? 監査ログは必要? |
| プライバシー・法規制 | 個人情報、コンプライアンス | 扱う個人情報は? 適用される法規制(GDPR/個情法等)は? |
| 運用性・可観測性 | ログ、監視、アラート | 何を監視する? 障害をどう検知する? |
| 保守性・拡張性 | 変更容易性 | 将来の拡張予定は? 誰が保守する? |
| ユーザビリティ・A11y | 使いやすさ、アクセシビリティ | 対象デバイス・ブラウザは? WCAG 準拠レベルは? |
| 互換性・移植性 | 既存連携、環境 | 連携する既存システムは? 対応環境は? |
| 国際化 | 多言語、地域 | 対応言語・通貨・タイムゾーンは? |
| コスト | 予算、運用費 | インフラ・外部サービスの予算上限は? |

## 曖昧さを潰す質問バンク

要望に出がちな曖昧語を、検証可能な条件に変換する問い。

| 曖昧語 | 聞くこと |
| :--- | :--- |
| 「速く」 | 目標レイテンシ/スループットの数値は? どの操作? |
| 「大量」「たくさん」 | 想定件数・データ量・同時実行数は? 増加率は? |
| 「ユーザー」 | どのロール? 認証済み/未認証? 権限の違いは? |
| 「使いやすく」 | 何を以て達成とみなす? 対象デバイスは? |
| 「リアルタイム」 | 許容遅延は? 秒? ミリ秒? |
| 「安全に」 | 何から守る? 認証/暗号化/監査のどれ? |
| 「すべて」「いつでも」 | 本当に全件/全時間? 例外は? 上限は? |

さらに各要件で確認する定番:
- **スコープ境界** — どこまでが今回? やらないことは?
- **エラー時の振る舞い** — 失敗したらどう見える? どこまで自動回復?
- **データのライフサイクル** — 保持期間は? 削除条件は? 論理/物理削除?
- **権限と可視性** — 誰が見える/操作できる? テナント分離は?
- **上限・制約** — 件数/サイズ/回数の上限は? レート制限は?
- **同時実行** — 同じデータを複数人が同時に触ったら?

## 受入基準の書き方(Given-When-Then)

「前提 / 操作 / 期待結果」で、検証可能に書く。

- **Given**(前提) — テスト開始時の状態・データ
- **When**(操作) — ユーザー/システムの行動
- **Then**(期待) — 観測可能な結果(画面・データ・副作用)

良い例:
> Given 在庫1の商品がカートにある / When 購入を確定する / Then 注文が作成され、在庫が0になり、確認メールが届く

避けたい例(検証不能・実装依存):
> 購入処理がちゃんと動く / 在庫テーブルの updateStock() が呼ばれる

ポイント:
- 1基準1観点。Happy だけでなく Error/境界も書く(`edge-case-finder` と連動)
- 副作用(DB状態・送信メール・ログ)まで期待結果に含める
- 実装の中身(関数名・SQL)ではなく、外から観測できる結果で書く

## 粒度と優先度のフレーム

**INVEST**(良いユーザーストーリーの条件):
Independent / Negotiable / Valuable / Estimable / Small / Testable

**MoSCoW**(優先度):
- Must — これが無いとリリースの意味がない
- Should — 重要だが、無くても代替/後回し可能
- Could — あれば価値、コストが低ければ入れる
- Won't(this time) — 今回はやらない。**非目的として明記**し、将来候補に記録
