## 内訳(明細)の作り方

### ロール別に積む(受託開発の定番)

| ロール | 単価の目安の置き方 |
| :--- | :--- |
| PM / PL | 管理工数。全体の10〜20%程度を別建てにすることが多い |
| アーキ / 上級SE | 設計・難所。単価高め |
| SE / PG | 実装の主力 |
| QA / テスター | テスト工程 |

`effort-estimate` の人日内訳を、ロール単価でこの表に流し込むと根拠が一貫する。

### 直接費は工数と分けて計上

- ソフトウェアライセンス、SaaS/クラウド利用料、機器、外注費、交通費 など
- 「式」「一式」で1行にするか、内訳を別表にする
- 原価(仕入れ)に対し販売価格を載せる場合、マージンの方針を社内で持っておく(見積書には販売価格のみ記載が通例)

### 数量×単価(パラメトリック)で見せる場合

| 品目 | 数量 | 単位 | 単価 |
| :--- | --: | :--- | --: |
| 画面開発 | 12 | 画面 | ○○ |
| API 開発 | 8 | 本 | ○○ |
| 帳票開発 | 3 | 帳票 | ○○ |

顧客にとって「何にいくら」が分かりやすい。数量根拠は要件と紐づける。

## 端数・税・値引きの扱い

- **計算順序を固定する**: 明細金額(税抜) → 小計 → 値引き → 課税対象額 → 消費税 → 税込合計
- **端数処理を明記**: 例「明細金額は円未満切り捨て、消費税は課税対象額に税率を乗じ円未満切り捨て」。切り捨て/四捨五入/切り上げを決め打ちで書く
- **消費税**: 税率(10%)を明記。軽減税率・非課税・不課税が混在する場合は区分ごとに小計を分ける
- **値引きは課税前に1行**: 「お値引き ▲○○」。明細単価を下げて調整すると、後で根拠が追えなくなる
- **税込/税抜を取り違えない**: 明細は税抜統一、合計は税込を太字で。総額表示が求められる相手には税込を主見出しにする

## 概算見積もり vs 確定見積もり

| | 概算見積もり | 確定見積もり |
| :--- | :--- | :--- |
| 目的 | 予算感・初期提案 | 契約・発注の根拠 |
| 前提 | 要件が粗い段階 | 要件・仕様が固まった段階 |
| 表現 | 「概算」「±○%」を明記、レンジ可 | 単一金額、前提を厳密に |
| 注意 | 確定値と誤解させない | 前提逸脱時の追加見積もり条件を明記 |

概算では `effort-estimate` のレンジ(最小〜最大)をそのまま「○○万円〜○○万円」と見せてもよい。

## 注意点・トラブル予防

- **前提条件はトラブル予防の最重要項目**。スコープ外・修正回数・環境・依存を書かないと「これも含まれるはず」で揉める
- **有効期限を必ず入れる**。人員確保・単価・為替の前提が時間で変わる
- **見積もり根拠を社内に残す**(`effort-estimate` の内訳・前提)。問い合わせや再見積もりに即応できる
- **送付前にレビュー**: 数量・単価・小計・税の再計算、前提とスコープの整合、機密情報(原価・社内メモ)の混入チェック(`pii-check` / 社内規定)
- **再見積もりの条件を書く**: 要件変更・スコープ追加・前提逸脱時は再見積もり、と明記しておく
