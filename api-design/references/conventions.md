# API 設計 規約チートシート

`api-design` skill 本体から呼び出される、詳細規約のリファレンス。
チームに社内標準があればそちらを優先し、無い場合の妥当なデフォルトとして使う。

## HTTP メソッドのセマンティクス

| メソッド | 用途 | 安全 | 冪等 | リクエストボディ |
| :--- | :--- | :--- | :--- | :--- |
| GET | 取得 | ✓ | ✓ | なし |
| POST | 作成・非冪等な操作 | - | -(Idempotency-Key で補える) | あり |
| PUT | 全体置換 | - | ✓ | あり |
| PATCH | 部分更新 | - | -(設計次第) | あり |
| DELETE | 削除 | - | ✓ | 通常なし |

## HTTP ステータスコードの使い分け

| コード | 意味 | 使う場面 |
| :--- | :--- | :--- |
| 200 OK | 成功 | GET / PATCH / 同期処理の成功 |
| 201 Created | 作成成功 | POST でリソース作成。`Location` ヘッダ付与 |
| 202 Accepted | 受理(非同期) | 後で処理する。ジョブ ID を返す |
| 204 No Content | 成功・本文なし | DELETE 成功など |
| 400 Bad Request | 入力不正 | バリデーションエラー、形式不正 |
| 401 Unauthorized | 未認証 | トークン無し/無効 |
| 403 Forbidden | 認可なし | 認証済みだが権限不足 |
| 404 Not Found | 不在 | リソースが存在しない |
| 409 Conflict | 競合 | 重複作成、楽観ロック衝突、状態矛盾 |
| 422 Unprocessable | 意味的に処理不能 | 形式は正しいが業務ルール違反 |
| 429 Too Many Requests | レート超過 | スロットリング。`Retry-After` 付与 |
| 500 / 503 | サーバ側障害 | 想定外エラー / 一時的に利用不可 |

400 と 422 の使い分けはチームで統一する(形式エラー=400、業務ルール違反=422 が一例)。

## エラーエンベロープ

全エラーで同じ形を返す。`code` は機械可読(クライアントの分岐用)、`message` は人間向け、`details` はフィールド単位のエラー。

```json
{
  "error": {
    "code": "ORDER_ITEM_OUT_OF_STOCK",
    "message": "一部の商品が在庫切れです",
    "details": [
      { "field": "items[0].sku", "reason": "out_of_stock" }
    ],
    "requestId": "req_01H..."
  }
}
```

- `code` は `SCREAMING_SNAKE_CASE` で安定させる(メッセージ文言は変えてもコードは変えない)
- `requestId` を入れるとサポート・ログ突合が楽になる
- スタックトレースや内部実装の詳細を返さない(情報漏洩)

## ページング方式

| 方式 | 仕組み | 向く場面 | 注意 |
| :--- | :--- | :--- | :--- |
| オフセット | `?limit=20&offset=40` | 件数が少なく、ページ番号が欲しい | 大量データで深いページが遅い・ズレる |
| カーソル | `?limit=20&cursor=<token>` | 大量データ、無限スクロール | ページ番号ジャンプ不可 |
| キーセット | `?limit=20&after_id=123` | ソートキーが安定している | ソート列の制約 |

大量・高頻度ならカーソル方式を推奨。レスポンスに `nextCursor`(または `hasMore`)を含める。

## 命名規約(デフォルト)

- パス: 名詞・複数形・ケバブケース(`/order-items`)。動詞は使わない
- 状態遷移など名詞で表しにくい操作は `POST /orders/{id}/cancel` のようにサブリソース化
- JSON フィールド: `camelCase` または `snake_case` をプロジェクトで統一(混在させない)
- 真偽値: `isActive` / `hasItems` のように肯定形で
- 日時: ISO 8601・UTC(`2026-06-14T08:30:00Z`)。フィールド名は `createdAt` 等

## バージョニング

| 方式 | 例 | 利点 | 欠点 |
| :--- | :--- | :--- | :--- |
| URL パス | `/v1/orders` | 明快、キャッシュしやすい | URL が変わる |
| ヘッダ | `Accept: application/vnd.api+json;version=1` | URL 不変 | 可視性が低い、扱いにくい |

公開 API は URL パス方式が無難。**破壊的変更**(フィールド削除・型変更・必須化・意味変更)のみ新バージョンを切り、後方互換な追加は同一バージョン内で行う。

## 冪等性

- GET / PUT / DELETE は本来冪等になるよう設計する
- POST(作成)は `Idempotency-Key` ヘッダを受け取り、同一キーの再送には最初の結果を返す
- 決済・注文など「二重実行が致命的」な操作では必須

## REST 以外

- **GraphQL**: エンドポイントは単一。スキーマ(型・クエリ・ミューテーション)が契約。N+1 とクエリ深さ制限、エラーは `errors[]` 配列に注意
- **gRPC**: `.proto` が契約。フィールド番号は再利用せず、削除は `reserved` に。後方互換のルールに従う
