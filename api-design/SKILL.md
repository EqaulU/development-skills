# API Design

実装に入る前に API の契約(エンドポイント・リクエスト/レスポンス・エラー・バージョニング)を設計し、クライアントとサーバの合意点を固める。

## いつ使うか

- 新規 API・公開 API・チーム間で共有する内部 API を作るとき
- 既存 API に後方互換を壊しうる変更を入れるとき
- フロント/バックを並行開発するため、先に契約を確定したいとき

`api-doc` は実装済みエンドポイント「から」OpenAPI を起こす下流の作業。`api-design` はその上流で、実装前に契約そのものを決める。

## 入力情報

開始時に把握する(不明なら質問):

- **ユースケース** — クライアントが何を達成したいか(画面・処理フロー)
- **リソース / ドメイン** — 扱う主要エンティティと関係
- **既存 API 規約** — 命名・バージョニング・エラー形式の社内標準
- **非機能要件** — 認証方式、想定負荷、レイテンシ、冪等性の要否
- **方式** — REST / GraphQL / RPC(gRPC 等)

## 設計の観点

詳細な規約(HTTP ステータス対応表、ページング方式、エラーエンベロープ、命名ルール)は [references/conventions.md](references/conventions.md) を参照。

1. **リソースモデリング** — 名詞中心。動詞を URL に入れない(REST)。ドメインの単位で切る
2. **命名** — 複数形/ケースの一貫性。社内標準に合わせる
3. **メソッドのセマンティクス** — GET は安全 / PUT・DELETE は冪等 / POST は非冪等
4. **リクエスト/レスポンス** — フィールド、必須・任意、型、null の扱い
5. **エラーモデル** — 統一エンベロープ、機械可読なコード、適切な HTTP ステータス
6. **ページング/フィルタ/ソート** — 大量データを返すエンドポイントの規約
7. **バージョニングと互換性** — 破壊的変更の定義と回避策
8. **認証認可** — 方式、スコープ、レート制限
9. **冪等性** — 再送安全性(POST の Idempotency-Key 等)

## 出力フォーマット

```markdown
## API 設計: 注文 (Orders)

### リソース
- **Order** — 注文。`id` / `status` / `items[]` / `totalAmount` / `createdAt`
- **OrderItem** — 注文明細(Order に内包)

### エンドポイント
| メソッド | パス | 用途 | 冪等 | 認証 |
| :--- | :--- | :--- | :--- | :--- |
| GET | /v1/orders | 一覧(ページング・フィルタ) | ✓ | 必須 |
| POST | /v1/orders | 作成 | Idempotency-Key | 必須 |
| GET | /v1/orders/{id} | 取得 | ✓ | 必須 |
| PATCH | /v1/orders/{id} | 部分更新 | - | 必須 |
| POST | /v1/orders/{id}/cancel | キャンセル(状態遷移) | - | 必須 |

### POST /v1/orders
**リクエスト(body)**
| フィールド | 型 | 必須 | 説明 |
| :--- | :--- | :--- | :--- |
| items | OrderItem[] | ✓ | 1件以上 |
| items[].sku | string | ✓ | 商品コード |
| items[].quantity | int | ✓ | 1以上 |
| couponCode | string | - | クーポン |

ヘッダ: `Idempotency-Key`(再送時の二重作成防止)

**レスポンス**
- `201 Created` — 作成された Order を返す。`Location: /v1/orders/{id}`
- `400` 入力検証エラー / `409` 在庫不足 / `422` クーポン不正

### エラーモデル
全エラーは統一エンベロープで返す(`code` は機械可読、`message` は人間向け)。
形式と HTTP ステータスの対応は references/conventions.md を参照。

### ページング
`GET /v1/orders?limit=20&cursor=...` のカーソル方式を採用(理由: ...)。

### バージョニングと互換性
- バージョン: URL パス `/v1`
- 破壊的変更(フィールド削除・型変更・必須化)は新バージョンで。既存 v1 は維持
- 後方互換な追加(任意フィールド追加)は v1 内で実施可
```

## 原則

- **契約を先に固める。** 実装の都合でレスポンス形が決まると、クライアントが振り回される。先に契約、後に実装。
- **破壊的変更を明示する。** フィールド削除・型変更・必須化は破壊的。新バージョンに分け、既存版は維持する。
- **エラーも設計対象。** 正常系だけでなく、エラーコード体系・ステータス・メッセージを最初から設計する。
- **一貫性 > 個別最適。** 命名・ページング・エラー形式は全エンドポイントで揃える。例外を作らない。
- **既存規約を尊重。** 社内の API 標準があればそれに合わせる。独自方式は規約が無いときだけ。
- **過剰設計を避ける。** 今のユースケースに不要なパラメータ・エンドポイントは作らない。
