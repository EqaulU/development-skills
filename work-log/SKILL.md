# Work Log (作業ログ)

コードや `git diff` には残らない「なぜその作業をしたか / 何に詰まってどう抜けたか / 次に何をするか」を、作業の区切りごとに時系列で残す。

## 何のために残すか

- **再開・引き継ぎを速くする。** 中断後や別の人が入るとき、文脈を一瞬で取り戻せる。
- **同じ轍を踏まない。** 詰まった点と解決をログに残せば、次に同じ問題が来たとき調べ直さずに済む。
- **「いつ何をしたか」を追える。** 後から経緯を問われたとき、コミットだけでは見えない判断の流れを補える。

ADR(重い設計判断)/ incident-postmortem(障害)/ changelog(リリース)の **手前にある、日々の作業の生ログ**。ここから重要なものを各成果物へ昇華する。

## いつ使うか

- 作業の区切り(着手・中断・完了)で記録を残したいとき
- 調査・試行錯誤の過程を後で振り返れるようにしたいとき
- 進捗メモ・デイリーログ・開発日誌を付けたいとき

1件の重い設計判断を残すなら `adr`、障害の事後分析なら `incident-postmortem`。work-log はそれより粒度の細かい、日々の作業ログ。

## 保存場所と粒度

| 運用 | パス例 | 向くケース |
| :--- | :--- | :--- |
| 日次(推奨) | `docs/worklog/YYYY-MM-DD.md` | 1日に複数タスク。1ファイルにエントリを追記していく |
| タスク単位 | `docs/worklog/YYYY-MM-DD-<topic>.md` | 1タスクが数日にまたがる/独立して参照したい |

推奨は **日次ファイルに追記**。見出しを「時刻 + タスク名 + 状態」にし、上から時系列で積む。既存ファイルがあれば末尾に追記(過去分は書き換えない)。

## 入力情報

開始時に把握する(不明なら会話履歴・`git diff`・`git log` から拾う):

- **やったこと** — 実装/調査/対応した内容
- **なぜ** — 背景・目的・対応 Issue / チケット
- **詰まった点と解決** — 問題 → 原因 → どう抜けたか
- **次のアクション** — 残タスク・宿題

## 出力フォーマット

1エントリ = 1作業の区切り。日次ファイルにこれを追記していく。

```markdown
## YYYY-MM-DD HH:MM <タスク名> [着手 / 進行中 / 完了 / 中断]

- 関連: #1234 / ブランチ `feature/xxx` / ADR-0007
- 目的: なぜこの作業をするか(1-2文)

### やったこと
- 実装・調査・対応した内容を事実ベースで箇条書き

### 詰まった点 / 判断
- 問題 → 原因 → 解決(または回避策)
- 迷った点とその場の判断。重い判断は `adr` に切り出してここからリンク

### 変更
- `path/to/file` — 何をどう変えたか
- 主要コミット: `abc1234`

### 結果 / 確認
- テスト・動作確認の結果(通った/落ちた、何を確認したか)

### 次のアクション
- [ ] 次にやること(担当・期限があれば付ける)
- [ ] 残った懸念・宿題
```

省略可: 単純な作業なら「やったこと」「次のアクション」だけでもよい。詰まりが無くても **次のアクションは必ず1つ** 書く。

## 記入のコツ / 原則

- **その場で書く。** 後追いだと「なぜ」「どう抜けたか」を思い出せない。区切りごとに即記録。
- **詰まりは必ず残す。** 解決した瞬間が一番価値が高い。原因と解決をセットで書く。
- **事実と推測を分ける。** 「〜のはず」「未検証」を明記。確認済みと未確認を混ぜない。
- **次のアクションを必ず1つ。** 次に開いたとき、ゼロから思い出さずに再開できる状態にする。
- **検索しやすく。** タスク名・キーワード・Issue 番号を見出しや本文に入れる。
- **機密・PII を書かない。** トークン・パスワード・個人情報は残さない(`pii-check` の観点)。
- **過去エントリは書き換えない。** 追記運用。訂正は新しいエントリで。

## 関連スキル

- `adr` — 作業中に下した重要な設計判断は ADR に切り出し、work-log からリンクする
- `task-breakdown` — 分解したサブタスク単位で作業ログのエントリを立てる
- `commit-msg` / `pr-description` — 作業ログのサマリをコミット本文・PR 説明に転記できる
- `changelog-writer` — リリース時、作業ログから利用者向けの変更を抽出する
- `incident-postmortem` — 障害対応中の作業ログはタイムライン・原因分析の素材になる
